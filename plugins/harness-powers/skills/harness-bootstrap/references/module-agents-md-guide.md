# Module-Level AGENTS.md Guide

Module-level AGENTS.md files define the boundary, patterns, and testing approach for a specific module or package within the repository. They exist to prevent agents from accidentally violating module boundaries or misunderstanding ownership.

## When to Create One

Create a module AGENTS.md when ANY of these is true:

- The module has its own build config (its own `package.json`, `pom.xml`, `Cargo.toml`)
- The module has >20 source files AND a distinct domain boundary
- The module already has its own README

Do NOT create one for:

- Utility directories with <5 files
- Test directories (they follow the module they test)
- Config directories
- Generated code directories

## Template

```markdown
<!-- Generated by harness-bootstrap. Safe to edit. Audited by harness-garden. -->
# <Module Name>

> Part of [<project-name>](../../AGENTS.md).

## Ownership

This module owns: <what this module is responsible for>

This module does NOT own: <explicit boundary — what belongs elsewhere>

## Key Patterns

- <Pattern 1: specific to this module>
- <Pattern 2: specific to this module>
- <Pattern 3: specific to this module>

## Testing

- Test location: `<relative path to tests>`
- Run tests: `<command to run only this module's tests>`
- Test style: <describe the testing approach — unit tests, integration tests, fixtures, etc.>

## Dependencies

- Depends on: <list modules this module imports from>
- Depended on by: <list modules that import from this module>
```

## Example: Payment Module

```markdown
<!-- Generated by harness-bootstrap. Safe to edit. Audited by harness-garden. -->
# Payment Module

> Part of [order-service](../../AGENTS.md).

## Ownership

This module owns: payment processing, refund handling, payment provider integration (Stripe, PayPal), and payment status tracking.

This module does NOT own: order lifecycle (that's `src/order/`), user billing history (that's `src/billing/`), or invoice generation (that's `src/invoice/`).

## Key Patterns

- All payment provider calls go through `PaymentGateway` interface — never call provider APIs directly
- Payment state transitions follow the state machine in `PaymentStatus.java` — no ad-hoc status changes
- Monetary amounts use `BigDecimal` with `HALF_UP` rounding, never `double` or `float`
- All provider webhooks are idempotent — use `idempotencyKey` for deduplication

## Testing

- Test location: `src/test/java/com/example/payment/`
- Run tests: `./gradlew test --tests "com.example.payment.*"`
- Test style: unit tests with Mockito for service layer, @SpringBootTest for integration with embedded H2

## Dependencies

- Depends on: `src/common/` (shared types), `src/config/` (payment provider config)
- Depended on by: `src/order/` (calls `PaymentService.processPayment()`)
```

## Sizing Guidelines

- **Target: 30-50 lines.** If it's growing past 50 lines, either the module is too large (consider whether it should be split) or you're including too much detail (move it to a design doc).
- **Boundary definition > implementation detail.** The most valuable information in a module AGENTS.md is what the module owns and what it doesn't. This prevents agents from putting code in the wrong place.
- **Patterns should be module-specific.** If a pattern applies to the whole repo, it belongs in the root AGENTS.md or `docs/ARCHITECTURE.md`, not in every module file.

## Placement

Module AGENTS.md files go at the root of the module directory:

```
src/
├── order/
│   ├── AGENTS.md          ← here
│   ├── OrderService.java
│   └── ...
├── payment/
│   ├── AGENTS.md          ← here
│   ├── PaymentService.java
│   └── ...
```

For monorepos with top-level packages:

```
packages/
├── web-app/
│   ├── AGENTS.md          ← here
│   ├── package.json
│   └── src/
├── api-server/
│   ├── AGENTS.md          ← here
│   ├── package.json
│   └── src/
```

The agent reads the nearest AGENTS.md in the directory tree, so the closest one takes precedence. Module AGENTS.md files supplement (not replace) the root AGENTS.md.
