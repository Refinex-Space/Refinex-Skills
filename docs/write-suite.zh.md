# Write Skills 套件说明

本套件由两个配套 Skill 组成，设计目标是无论起点如何，都能产出标准一致、质量相当的高质量技术文档。无论你是从空白页面创作，还是基于低质量的现有材料重建，两个 Skill 执行相同的写作标准，使最终产物在质量、叙述声音和文档结构上无法被区分出来源差异。

---

## 两个 Skill 的定位

### `tech-writing` — 从零创作

当你有明确的主题、技术论点或需要记录的技术决策，但没有现成材料可以参考时，使用这个 Skill。

Skill 在生成任何正文之前，强制执行一个**写前协议**：明确中心论点（不是话题，是可辩护的声明）、收集具体的技术锚点（真实数字、失败模式、被拒绝的方案）、审计读者的既有认知、声明范围边界、选择叙述声音。只有在这个简报完成之后，写作才会开始。

这个 Skill 对抗的核心风险是**锚点饥饿**——AI 在没有具体技术基础的情况下会自动滑向百科全书式的输出：描述事物是什么，但从不对它们意味着什么作出判断。

**触发场景：** 写一篇关于 X 的技术博客、为 Y 起草架构文档、对比 X 和 Y、写一份 ADR、深入解析 X 的工作原理、为 Z 编写 API 文档。

---

### `tech-rewrite` — 基于现有材料重建

当你手头有任何质量的现有材料——内部笔记、会议纪要、AI 生成的草稿、遗留 Wiki、代码注释、早期版本文档——需要将其转化为高质量技术文档时，使用这个 Skill。

Skill 在提取阶段和写作阶段之间强制建立严格隔离。源材料绝不作为写作模板使用——它被当作原始情报处理：所有事实被提取进结构化的 Fact Register，模糊声明被显式丢弃，质量问题被系统诊断，污染风险被逐一命名，然后才开始写第一个字。

这个 Skill 对抗的核心风险是**风格污染**——七种具体的污染机制，使低质量源材料在写作者有意改进的情况下依然腐蚀输出质量。这七种机制分别是：结构镜像、虚空继承、模糊性漂白、语气渗透、决策理由缺失、虚假完整性、范围膨胀。

**触发场景：** 重写这份文档、整理我的笔记、这份草稿质量很差帮我修好、把这些内容变成一篇正式文章、基于这份材料写一篇关于 X 的文档、改进这份文档。

---

## 共享质量标准

两个 Skill 产出的文档受同一套质量规则约束。一篇由 `tech-rewrite` 重建的文档和一篇由 `tech-writing` 从零创作的同主题文档，质量上应当无法被区分。

**结构标准。** 每篇文档都有一个携带论点的标题（而不只是话题标签）、包含范围说明和前置知识的文档头信息块、以及在读者阅读开始 60 秒内即亮明中心论点的开篇段落。每个对比章节以明确的裁决结束。每个架构或设计决策都包含被拒绝的替代方案及其原因。

**深度标准。** 每篇文档都包含一个写明具体机制（而非泛泛类别）的失败模式章节，以及一个界定建议适用范围的局限性章节。高级工程师测试强制执行：如果某个章节没有教给读者任何官方文档里没有的内容，它要么被深化，要么被删除。

**叙述声音标准。** 四种叙述声音被明确定义（Production War Story、Design Tribunal、Mechanism Autopsy、Migration Field Guide），每篇文档选择一种并全程保持一致。AI 味道模式——假平衡、空洞超级词、背景堆砌、被动式责任规避——被显式禁止，并附有具体的识别和修正规则。

**语言标准。** 中文技术写作：技术专有名词保持英文，语气目标是高级工程师在设计评审中向同事解释难题的方式（不官僚，不随意），结论前置（论点在先，证据在后）。英文写作：优先选用盎格鲁-撒克逊词汇，精确性优先于简洁性。

## 推荐使用模板

以下是各主要文档类型的推荐起始模板。复制相关模板，填写括号内的内容，提交给对应的 Skill。

---

### 模板 A — 技术博客 / 深度分析文章
*适用 Skill：`tech-writing`*

```
写一篇技术博客文章。

主题：[具体的技术、决策或机制]

中心论点：[一句话——文章将捍卫的声明，不是话题本身]
示例："在以 SSE 为主的 AI 推理服务中，WebFlux 不只是一个选项——
它是唯一能在规模化场景下避免线程池耗尽的架构。"

目标读者：
- 职位：[例如 Java 后端工程师、平台架构师]
- 经验级别：[初级 / 中级 / 高级]
- 已知前置知识：[列出读者已知的内容]
- 他们目前可能持有的认知：[文章可能挑战或确认的既有假设]

深度要求：[表面 / 标准 / 深度]
- 表面：解释是什么和为什么，不涉及实现细节
- 标准：包含关键实现模式和配置示例
- 深度：内部机制分析、失败模式解剖、生产级配置及选择理由

必须覆盖的内容（每项都需要给出判断，不只是描述）：
1. [内容点 1]
2. [内容点 2]
3. [内容点 3]

明确不覆盖的内容：
- [排除项]

语言：[中文 / English]
```

### 模板 B — 架构决策记录（ADR）
*适用 Skill：`tech-writing`*

```
写一份架构决策记录。

决策标题：[动词短语——例如"在 AI 推理服务层采用 WebFlux"]

背景：
[描述具体问题和约束条件。不做决策会发生什么？
有哪些不可协商的要求？2–4 句话。]

决策内容：[一句话说明选择的方案]

考虑过的替代方案：
- [方案 1]：[为何考虑，为何拒绝]
- [方案 2]：[为何考虑，为何拒绝]

影响本次决策的关键约束条件：
- [约束 1：例如"必须在 4 vCPU 上支撑 500 个并发 SSE 连接"]
- [约束 2]

需要记录的后果：
- 正面影响：[这个决策开启了什么]
- 负面影响：[这个决策关闭了什么——请诚实记录]
- 风险：[哪些条件变化会使这个决策在未来被证明是错误的]

语言：[中文 / English]
```

### 模板 C — 模块 / 组件设计文档
*适用 Skill：`tech-writing`*

```
写一份模块设计文档。

模块名称：[名称]
范围：[一句话——这个模块做什么，明确不做什么]

目标读者：[将要实现或扩展这个模块的工程师职位和经验级别]

设计约束条件（不可更改的需求，决定了设计方向）：
- [约束 1：例如"必须是无状态的——请求之间无本地状态"]
- [约束 2]

需要记录的关键设计决策（每项都包含被拒绝的替代方案）：
1. [决策 1]
2. [决策 2]

需要记录的接口契约：
- 公开 API：[方法签名或端点规范]
- 发出的事件：[如适用]
- 依赖关系：[这个模块对其依赖项的要求，以契约形式表达]

需要覆盖的失败模式：
- [失败场景 1]
- [失败场景 2]

语言：[中文 / English]
```

### 模板 D — 技术选型对比指南
*适用 Skill：`tech-writing`*

```
写一份技术选型对比指南。

选型场景：[使这次对比成为必要的具体情境]

可接受最终一致性。"

对比选项：
- 选项 A：[名称 + 一句话说明它是什么]
- 选项 B：[名称 + 一句话说明它是什么]

评估标准（按本场景重要性排序）：
1. [标准 1：例如"1000 并发请求下的读取延迟"]
2. [标准 2：例如"无专职基础设施工程师的团队的运维复杂度"]
3. [标准 3]

我的预判（希望被挑战或确认）：
[例如"我认为 Redis 是显而易见的选择，但我想知道是否存在
Memcached 足够有优势、值得分拆运维的场景。"]

明确不覆盖的内容：[排除项]
语言：[中文 / English]
```

### 模板 E — 基于现有材料的重写 / 重建
*适用 Skill：`tech-rewrite`*

```
重写以下文档。

输出的文档类型：[技术博客 / 架构文档 / ADR / 模块设计 / API 参考]

输出的目标读者：
- 职位：[例如不熟悉响应式编程的中级 Java 工程师]
- 阅读后应具备的能力：[一句话——具体的能力或决策目标]

我已知的源材料问题（可选，但填写后有助于提高输出质量）：
- [例如"完全没有失败模式章节"]
- [例如"选择 Kafka 的理由从未解释"]
- [例如"文档结构照搬了内部工单格式，不符合阅读逻辑"]

需要保留的约束条件：
- [例如"技术选型是最终决定——不要质疑它们，只需解释它们"]
- [例如"篇幅必须控制在 1500 字以内"]

希望补充的内容：
- [例如"补充失败模式章节——我知道主要有 X 和 Y 两种情况"]

输出语言：[中文 / English]

---

[在此粘贴源材料]
```

## Skill 选择速查表

| 起点 | 目标文档类型 | 使用 Skill |
|---|---|---|
| 有主题和观点，无现有材料 | 博客、深度分析 | `tech-writing` |
| 需要记录一个架构决策 | ADR | `tech-writing` |
| 需要设计并记录一个新模块 | 模块设计文档 | `tech-writing` |
| 需要为技术选型提供依据 | 对比指南 | `tech-writing` |
| 现有文档存在质量问题 | 任意类型 | `tech-rewrite` |
| AI 生成的草稿需要改进 | 任意类型 | `tech-rewrite` |
| 会议纪要 / 内部 Wiki | 博客、设计文档 | `tech-rewrite` |
| 代码注释 + 散落笔记 | 模块文档、ADR | `tech-rewrite` |
| 现有材料存在但不完整 | 任意类型 | `tech-rewrite` |

**选择原则：** 如果你有现成的书面材料，使用 `tech-rewrite`。如果你从一个主题和一个观点出发，使用 `tech-writing`。

---

## 关于质量一致性的说明

两个 Skill 共享同一套写作标准而不是各自定义，原因在于：质量应当是输出的属性，而不是生产方法的属性。一篇成品文档的读者不应能判断出它是从零创作的还是从低质量材料重建的。如果重建的痕迹可以被感知——如果它读起来像是某个东西的"整理版"而不是一篇一流的独立文档——说明 `tech-rewrite` 的流程没有被正确执行，应当重新检查污染模式参考文件后再推进。
