## Design Principles

### The Map, Not the Territory

AGENTS.md is a navigation aid. It tells the agent where to look, not everything it needs to know. The actual knowledge lives in the code, the tests, the config files, and the dedicated documentation. AGENTS.md connects these pieces.

Think of it like a well-organized library: the card catalog tells you which shelf to check, but you still read the actual book. A card catalog that tries to summarize every book defeats its own purpose.

### Progressive Disclosure

An agent starting work in a subdirectory should get exactly the context it needs for that scope — no more, no less. Information is layered:

- **Layer 0 (always loaded):** Root AGENTS.md — project identity, stack, hard invariants, architecture map
- **Layer 1 (scope-loaded):** Module AGENTS.md — patterns, conventions, and rules specific to this module
- **Layer 2 (on-demand):** Referenced docs — architecture decisions, API specs, design docs, read only when needed

### Verifiability

Every statement in an AGENTS.md should be mechanically or manually verifiable. "We use the repository pattern" is verifiable (grep for it). "We value clean code" is not. Prefer the former.

### Freshness Over Completeness

A shorter, accurate AGENTS.md is worth more than a comprehensive but stale one. If you're uncertain whether something is still true, omit it and add a TODO marker. Stale instructions are worse than no instructions — agents follow them with confidence.

## Root AGENTS.md Template

The root file should be 80–120 lines. Here is the structural template:

```markdown
# Project Name

One-sentence description of what this project does and who it serves.

## Tech Stack

- **Language:** TypeScript 5.x (strict mode)
- **Framework:** Next.js 15 (App Router)
- **Database:** PostgreSQL 16 via Prisma ORM
- **Testing:** Vitest + Playwright
- **CI:** GitHub Actions
- **Package Manager:** pnpm 9.x

## Architecture

Brief (3-5 sentence) description of the high-level architecture. Name the major
domains and their relationships.

See `docs/architecture.md` for the full architecture map.

### Dependency Direction

Types → Config → Repository → Service → Runtime → UI

Imports must follow this direction. A Service may import from Repository but never
the reverse. Enforced by [eslint-plugin-import/no-restricted-paths | custom linter].

### Module Map

| Directory        | Purpose                        | AGENTS.md |
|-----------------|--------------------------------|-----------|
| `src/api/`      | REST API routes and middleware | ✓         |
| `src/core/`     | Domain models and business logic| ✓        |
| `src/infra/`    | Database, cache, external APIs | ✓         |
| `src/frontend/` | React components and pages     | ✓         |
| `tests/`        | Test suites and fixtures       | ✓         |

## Hard Invariants

These rules must never be violated. They exist because violations have caused
real production incidents or severe quality degradation.

1. All API responses use the `ApiResponse<T>` wrapper from `src/core/types/api.ts`
2. Database queries go through the repository layer — no raw SQL in services
3. Environment variables are accessed only via `src/config/env.ts`, never `process.env` directly
4. All user-facing strings go through the i18n system — no hardcoded text
5. Feature flags are checked via `src/core/flags.ts`, never by environment checks

## Build & Test Commands

```bash
pnpm install          # Install dependencies
pnpm dev              # Start dev server (port 3000)
pnpm build            # Production build
pnpm test             # Run unit + integration tests
pnpm test:e2e         # Run Playwright E2E tests
pnpm lint             # ESLint + Prettier check
pnpm lint:fix         # Auto-fix lint issues
pnpm db:migrate       # Run pending migrations
pnpm db:seed          # Seed development data
```

## Conventions

- **Branching:** `feature/<ticket>-<short-desc>`, `fix/<ticket>-<short-desc>`
- **Commits:** Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`)
- **PRs:** Must pass CI, require 1 review, squash-merge to main
- **Error handling:** Use `Result<T, E>` for expected failures, throw only for programmer errors

## Key Documentation

- `docs/architecture.md` — System architecture and domain boundaries
- `docs/conventions.md` — Detailed coding conventions
- `docs/adr/` — Architecture Decision Records
- `CONTRIBUTING.md` — Contribution workflow
```

## Module-Level AGENTS.md Template

Module files should be 30–80 lines. They add only what's specific to this scope.

```markdown
# API Module (`src/api/`)

REST API layer. Handles HTTP routing, request validation, authentication
middleware, and response serialization.

## Patterns

### Route Organization
Routes are grouped by domain in subdirectories: `src/api/auth/`, `src/api/billing/`,
`src/api/users/`. Each domain directory contains:
- `router.ts` — Route definitions (Express Router)
- `handlers.ts` — Request handler functions
- `validators.ts` — Zod schemas for request validation
- `__tests__/` — Handler tests

### Request Flow
1. Express router matches the route
2. Auth middleware validates the JWT (except public routes listed in `config/public-routes.ts`)
3. Validation middleware runs the Zod schema against the request body
4. Handler calls the appropriate service method
5. Handler wraps the result in `ApiResponse.success()` or `ApiResponse.error()`

### Error Handling
Handlers must never throw. Use `try/catch` and return `ApiResponse.error()` with
the appropriate HTTP status code. The global error handler (`src/api/middleware/error.ts`)
catches unhandled exceptions and returns 500.

## Pitfalls

- **Don't import from `src/frontend/`** — The API layer has no dependency on frontend code.
- **Don't access `req.user` without auth middleware** — Routes without `requireAuth()`
  will have `req.user` as undefined.
- **Don't return raw Prisma objects** — Always map through a DTO. Prisma models
  may contain fields that shouldn't be exposed (e.g., `passwordHash`).

## Testing

Tests use `supertest` against the Express app. Each handler gets at least:
- Happy path test
- Validation failure test (malformed input)
- Auth failure test (no token / expired token)

Test fixtures are in `tests/fixtures/api/`. Use `createTestUser()` from
`tests/helpers/auth.ts` for authenticated requests.
```

## Content Calibration Examples

### Good: Specific and Actionable
```markdown
## Error Handling
Use `Result<T, AppError>` for all service methods. The `AppError` type is in
`src/core/errors.ts` and includes `code`, `message`, and optional `details`.
API handlers unwrap Results and map AppError codes to HTTP status codes via
the mapping in `src/api/middleware/error-mapper.ts`.
```

### Bad: Vague and Unverifiable
```markdown
## Error Handling
Handle errors properly. Use appropriate error types and make sure errors
are logged correctly.
```

### Good: Concise Hard Invariant
```markdown
5. Feature flags are checked via `src/core/flags.ts`, never by environment checks
```

### Bad: Over-Explained Invariant
```markdown
5. We have a sophisticated feature flag system that we built to manage gradual
   rollouts and A/B testing. It's important that everyone uses the centralized
   system located at `src/core/flags.ts` instead of checking environment variables
   directly, because the feature flag system provides additional benefits like
   logging, percentage-based rollouts, and user targeting. If you check environment
   variables directly you'll miss out on all of these benefits and create
   inconsistency in the codebase.
```
The good version says the same thing in one line. The agent doesn't need the history lesson.

### Good: Cross-Reference Pointer
```markdown
See `docs/adr/003-event-sourcing.md` for why we chose event sourcing over CRUD for the billing domain.
```

### Bad: Inlining the Reference
```markdown
We use event sourcing for billing because in Q2 2024 we had an incident where...
[300 words of context that belongs in the ADR]
```

## Hard Rules for All AGENTS.md Files

1. **Maximum line counts:** Root ≤ 120 lines. Module ≤ 80 lines. If you exceed this, you're including content that belongs in a referenced document.

2. **No aspirational content.** Every statement must describe the current state. If something is planned but not yet implemented, it does not belong in AGENTS.md. Put it in a roadmap doc and link to it.

3. **No duplicated content across levels.** If the root AGENTS.md says "we use Conventional Commits", module-level files must not repeat it. They may add module-specific commit conventions (e.g., "use `feat(api):` prefix for this module").

4. **Concrete commands, not descriptions of commands.** Write `pnpm test:e2e --project=api` not "run the E2E tests for the API module".

5. **Link, don't inline.** If context requires more than 3 sentences to explain, it belongs in a dedicated document. AGENTS.md should link to it.

6. **One voice, declarative.** Write in imperative or declarative voice. "All API responses use the wrapper" not "You should use the wrapper" or "We decided to use the wrapper".

7. **Machine-friendly structure.** Use consistent headings, tables, and code blocks. Agents parse structure. Avoid long paragraphs that bury actionable information in narrative prose.
