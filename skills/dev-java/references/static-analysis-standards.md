## 1. SonarQube Critical Rules

These are the SonarQube rules that appear most frequently in Java projects and carry the highest severity. All code generated by this skill should pass these rules.

**Cognitive Complexity**: Methods should not exceed a cognitive complexity of 15 (SonarQube default). Deep nesting, multiple conditions, and switch statements inflate cognitive complexity. Refactor by extracting methods, using early returns (guard clauses), or applying Strategy/State patterns.

**Duplicated Code**: No block of 10+ consecutive lines should be duplicated. Extract shared logic into utility methods or base classes. Duplicated code is not just a style issue — it is a maintenance landmine where a fix in one copy is missed in another.

**Unused Imports and Variables**: Dead code is noise. Remove all unused imports (IDE auto-cleanup). Remove all unused local variables, parameters, and private methods. If a parameter is unused but required by an interface contract, prefix it with an underscore or annotate with `@SuppressWarnings("unused")` with a comment.

**Security Hotspots**: Never hardcode passwords, API keys, or secrets. Never use `Math.random()` for security-sensitive operations — use `SecureRandom`. Never construct SQL with string concatenation — use parameterized queries. Never disable certificate validation in HTTP clients.

**Null Dereference**: Every object reference obtained from external input (method parameters, database results, HTTP responses, configuration values) must be null-checked or guaranteed non-null by contract (`@NonNull`, `Optional`).

## 2. Alibaba Java Coding Guidelines Alignment

The Alibaba guidelines (p3c) categorize rules into Mandatory, Recommended, and Reference levels. The following Mandatory rules are non-negotiable.

**Naming**: Names must not start or end with underscores or dollar signs. Abstract classes must be named `AbstractXxx` or `BaseXxx`. Exception classes must end with `Exception`. Test classes must end with `Test`. Type parameters: `T` for type, `E` for element, `K`/`V` for key/value, `N` for number.

**Constants**: Do not use magic values directly in code. Define them as named constants in a constants class or an enum. Magic values include numbers, strings, and characters whose meaning is not self-evident from context. `0`, `1`, `""`, and `null` are borderline — use named constants when their meaning is domain-specific.

**OOP Rules**: Use `@Override` on every overriding method — this catches signature mismatches at compile time. Do not use deprecated classes or methods in new code. Varargs (`Object...`) should only be used in the method with the most parameters when overloading. The `equals` method of `Object` must be used to compare object references; `==` is only for primitives and enums.

**Concurrent Programming**: Thread names assigned via `ThreadFactory` must be meaningful — never use default names like `pool-1-thread-1` in production. When creating `SimpleDateFormat`, declare it as a local variable (it is not thread-safe) or use `DateTimeFormatter` (immutable, thread-safe). Use `CountDownLatch` instead of `Thread.sleep()` for inter-thread coordination in tests.

**Collection Rules**: When converting between `Collection` and array, use `collection.toArray(new String[0])` — the zero-length array pattern triggers JVM optimizations. Do not remove/add elements during a `foreach` loop; use `Iterator` or `removeIf`. Check `isEmpty()` before accessing elements; never assume a collection is non-empty after a query.

**Date/Time**: Use `java.time` (Java 8+) for all new code: `LocalDate`, `LocalDateTime`, `ZonedDateTime`, `Instant`. Never use `Date`, `Calendar`, or `SimpleDateFormat` in new code. The `YYYY` pattern in `SimpleDateFormat` means "week year" (ISO 8601), not calendar year — use `yyyy`. This is a common production bug.

## 3. Lombok: Safe and Unsafe Usage

Lombok reduces boilerplate but can hide important design decisions. Follow these guidelines strictly.

**Safe to use (Recommended)**:
`@Slf4j` — generates a logger field. Clean and universally accepted.
`@RequiredArgsConstructor` — generates a constructor for `final` fields. Works well with Spring constructor injection.
`@Getter` on fields — transparent, no surprises.
`@Builder` — generates a fluent builder. Useful for DTOs with many fields.
`@ToString(exclude = {"password", "sensitiveField"})` — always exclude sensitive fields.
`@EqualsAndHashCode` — but read the caveats below.

**Use with caution**:
`@Data` — combines `@Getter`, `@Setter`, `@ToString`, `@EqualsAndHashCode`, `@RequiredArgsConstructor`. The problem: it generates setters (making objects mutable), and `@EqualsAndHashCode` includes all fields (dangerous for JPA entities with lazy-loaded associations). Prefer `@Value` (immutable) or explicit `@Getter` + `@RequiredArgsConstructor`.
`@EqualsAndHashCode` on JPA entities — NEVER include lazy-loaded `@ManyToOne` / `@OneToMany` fields. Always use `@EqualsAndHashCode(onlyExplicitlyIncluded = true)` and annotate only the business key fields with `@EqualsAndHashCode.Include`.
`@Setter` on entities — encourages anemic domain model. Prefer domain methods that enforce invariants (`order.cancel(reason)` instead of `order.setStatus(CANCELLED)`).

**Avoid**:
`@SneakyThrows` — silently swallows checked exceptions by wrapping them in `RuntimeException`. This breaks the exception contract and makes debugging harder. Explicitly declare or catch checked exceptions.
`@val` / `@var` — Java 10+ has native `var`. No need for Lombok's version.
`@Cleanup` — try-with-resources is clearer and standard. No need for Lombok here.
`@AllArgsConstructor` on entities — bypasses validation logic. Use `@Builder` with validation in the builder's `build()` method, or a static factory method.

## 4. Code Formatting Conventions

Formatting is the lowest-cost, highest-impact code quality measure. Non-negotiable rules:

**Indentation**: 4 spaces. No tabs. This is the Java ecosystem standard and the default in IntelliJ IDEA and Eclipse.

**Line length**: Maximum 120 characters. Break long lines at logical points — after a comma in parameter lists, before a `+` in string concatenation, after an arrow `->` in lambdas.

**Braces**: Always use braces for `if`, `else`, `for`, `while`, `do` — even for single-line bodies. The "goto fail" bug (Apple, 2014) happened because of a braceless if-statement. Opening brace on the same line as the statement (K&R style, Java convention).

**Blank lines**: One blank line between methods. Two blank lines between sections (fields, constructors, public methods, private methods). No blank line at the start or end of a block.

**Import ordering**: (1) `java.*`, (2) `javax.*`, (3) third-party (`com.*`, `org.*`), (4) project-internal. No wildcard imports (`import java.util.*`). Each class gets its own import. IDEs handle this automatically — enforce it in CI with Checkstyle.

**Method ordering within a class**: (1) Static fields, (2) Instance fields, (3) Constructors, (4) Public methods, (5) Protected methods, (6) Private methods, (7) Inner classes/enums. Within each group, order by logical relevance, not alphabetically.

## 5. SpotBugs / FindBugs Key Rules

**EI_EXPOSE_REP / EI_EXPOSE_REP2**: Never return mutable internal state directly. Return a defensive copy.

```java
// Bad — caller can modify internal list
public List<Item> getItems() { return items; }

// Good — defensive copy
public List<Item> getItems() { return Collections.unmodifiableList(items); }
```

**NP_NULL_ON_SOME_PATH**: Every path that could lead to a null dereference is flagged. Address by adding null checks, using `Optional`, or annotating with `@NonNull`/`@Nullable`.

**SE_BAD_FIELD**: Non-serializable fields in a `Serializable` class. Either make the field `transient` or ensure the type is serializable. In practice, avoid `Serializable` unless required by specific frameworks (HttpSession, JMS).

**RCN_REDUNDANT_NULLCHECK_OF_NONNULL_VALUE**: Checking for null after a method documented to never return null. Remove the redundant check — it confuses readers into thinking null is possible.

**DM_BOXED_PRIMITIVE_FOR_PARSING**: Use `Integer.parseInt()` instead of `Integer.valueOf().intValue()`. The latter creates an unnecessary boxed object.
