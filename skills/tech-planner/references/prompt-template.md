# Tech-Writing Prompt Template

## Purpose

This template defines the exact format for the self-sufficient writing prompts generated by tech-planner. Each prompt is the interface contract between tech-planner (upstream) and tech-writing (downstream). A user must be able to copy any single prompt into the tech-writing Skill weeks later with zero additional context and receive a high-quality article.

---

## Template

Every prompt must use exactly this structure. All fields marked [REQUIRED] are mandatory with no exceptions. Fields marked [CONDITIONAL] are included only when the stated condition applies.

```markdown
# Tech-Writing Prompt

## 系列上下文 [REQUIRED]
- **系列名称**: {系列名称}
- **当前位置**: 第 {N} 篇 / 共 {Total} 篇
- **系列总目标读者**: {目标读者画像的一句话描述}

## 文章标题 [REQUIRED]
{论点式中文标题}

## 中心论点 [REQUIRED]
{一句可证伪的判断。读者读完文章后应能判断这个论点是否成立。}

## 目标读者状态 [REQUIRED]
- **读者此刻知道什么**: {列出读者在读完前序文章后已经掌握的关键概念，具体到概念名称}
- **读者此刻不知道什么**: {列出本文将要填补的知识空白}
- **读者可能持有的误解**: {列出读者基于已有知识可能形成的错误推断}

## 关键技术锚点 [REQUIRED]
{列出需要用来支撑中心论点的具体技术证据。每个锚点应指明需要什么类型的真实素材。}
- 锚点 1: {描述} — 需要: {真实数据 / 源码分析 / 失败机制演示 / 被否方案对比 / 边界条件测试}
- 锚点 2: {描述} — 需要: {同上}
- ...

## 范围边界 [REQUIRED]
- **本文覆盖**: {明确列出本文讨论的内容范围}
- **本文不覆盖**: {明确列出本文不讨论的内容，以及不覆盖的原因——是因为将在后续文章处理，还是因为超出系列范围}

## 前序依赖 [REQUIRED]
{列出本文依赖的前序文章及其中的具体概念。如果是系列第一篇，写 "无前序依赖——本文是系列入口点。"}
- 依赖第 {X} 篇《{标题}》中的: {概念名称} — 可直接引用，无需重新解释
- ...

## 视觉说明计划 [REQUIRED]
{列出本文中需要图表辅助说明的机制或流程。}
- 图表 1: {描述要揭示什么} — 推荐类型: {Mermaid 图类型: flowchart / sequence / state / class / ER / timeline}
- 图表 2: ...
- 如果本文不需要图表，写 "本文以代码示例为主要说明手段，无需额外图表" 并简述原因。

## 深度契约 [REQUIRED]
{声明本文承诺达到的技术深度层次。选择一个或多个：}
- [ ] API 使用层面：教读者正确使用 API，理解参数含义和返回值
- [ ] 内部机制层面：解释 API 背后的工作原理，读者能预测边界行为
- [ ] 源码实现层面：深入关键源码路径，读者能阅读和调试框架代码
- [ ] 架构决策层面：分析设计权衡，读者能评估该技术是否适合自己的场景

## 写作约束 [REQUIRED]
- **目标字数**: {X-Y 字}
- **代码示例要求**: {需要 / 不需要 / 关键路径需要}。如需代码，指定语言和风格（完整可运行 vs 关键片段）。
- **性能数据**: {需要引用基准测试 / 不需要 / 需要定性讨论}
- **方案对比**: {需要与 {具体方案} 对比 / 不需要}

## 螺旋标注 [CONDITIONAL: 仅当本文重访了前序文章已介绍的概念时包含]
- **重访概念**: {概念名称}，首次出现于第 {X} 篇
- **本次新增复杂度**: {明确描述本次重访增加了什么新的复杂度层次，这是前序文章中未涉及的}

## 常见误解 [CONDITIONAL: 仅当本文主题存在广泛误解时包含]
{列出与本文主题相关的常见误解，帮助写作时主动纠正。}
- 误解 1: {描述} — 实际情况: {简述}
- ...

## 调研指引 [CONDITIONAL: 仅当有特别值得查阅的资源时包含]
{推荐 tech-writing Skill 在写作前查阅的特定资源。}
- {资源描述}: {URL 或搜索关键词}
- ...
```

## Field Filling Guidelines

### 中心论点
Must be a single falsifiable claim. Test: can someone read the article and conclude "this claim is correct" or "this claim is incorrect"?

**Good**: "Spring AI 的 Advisor 链机制在设计上优先保证了可观测性，但代价是对流式响应场景引入了不可忽视的延迟。"
**Bad**: "本文介绍 Spring AI 的 Advisor 链机制。" (Not falsifiable — it's a topic declaration.)

### 目标读者状态
The "读者此刻知道什么" field must list concrete concepts, not vague descriptions. Reference the specific article numbers where these concepts were introduced.

**Good**: "读者已理解 ChatClient 的 builder pattern（第 2 篇）、auto-configuration 的 Bean 注册时序（第 3 篇）。"
**Bad**: "读者对 Spring AI 有基本了解。"

### 关键技术锚点
Each anchor must specify the *type* of evidence needed. This tells the downstream writing skill what kind of research or demonstration to prepare.

Types of evidence:
- **真实数据**: Benchmark results, metrics, statistics from real-world usage
- **源码分析**: Specific source code paths that reveal the mechanism
- **失败机制演示**: A concrete scenario where the technology fails or behaves unexpectedly
- **被否方案对比**: An alternative approach that was considered but rejected, and why
- **边界条件测试**: A test at the edge of the technology's design envelope

### 视觉说明计划
Recommend specific Mermaid diagram types based on what needs to be shown:
- `flowchart`: Decision processes, data flow, algorithm steps
- `sequence`: Multi-component interactions over time, request/response cycles
- `state`: State machines, lifecycle transitions
- `class`: API hierarchies, interface relationships
- `ER`: Data model relationships
- `timeline`: Version evolution, migration paths

## Self-Sufficiency Checklist

Before finalizing each prompt, verify:

1. Could a writer produce the article with ONLY this prompt and web access, having never read the other articles in the series? (The "读者此刻知道什么" field must give enough context.)
2. Is the central thesis falsifiable?
3. Are all technical anchors specific enough that the writer knows what to search for or demonstrate?
4. Does the scope boundary clearly prevent scope creep while explaining *why* certain topics are excluded?
5. Does the visual plan tell the writer what each diagram should *reveal*, not just what it should *depict*?
6. If conditional fields apply, are they included?
