# Harness Manifest Schema

The harness manifest (`docs/generated/harness-manifest.md`) is the machine-readable inventory of all control plane artifacts. It serves two purposes:

1. **Verification target**: `scripts/check_harness.py` parses this file and checks that every listed artifact exists and is within staleness bounds.
2. **Audit record**: `harness-garden` reads this file to detect drift, update verification dates, and identify artifacts that need attention.

## Format

The manifest is a Markdown file with structured tables that can be parsed by simple regex patterns. This keeps it human-readable AND machine-parseable without requiring JSON or YAML.

## Template

```markdown
<!-- Generated by harness-bootstrap. Updated by harness-garden. -->
<!-- Do not delete entries manually — use harness-garden to manage this file. -->
# Harness Manifest

## Control Plane Artifacts

| Path | Type | Created | Last Verified |
|------|------|---------|---------------|
| AGENTS.md | root-agents | YYYY-MM-DD | YYYY-MM-DD |
| docs/PLANS.md | managed-doc | YYYY-MM-DD | YYYY-MM-DD |
| docs/ARCHITECTURE.md | managed-doc | YYYY-MM-DD | YYYY-MM-DD |
| docs/OBSERVABILITY.md | managed-doc | YYYY-MM-DD | YYYY-MM-DD |
| docs/exec-plans/active/ | directory | YYYY-MM-DD | YYYY-MM-DD |
| docs/exec-plans/completed/ | directory | YYYY-MM-DD | YYYY-MM-DD |
| docs/exec-plans/tech-debt-tracker.md | managed-doc | YYYY-MM-DD | YYYY-MM-DD |
| docs/generated/harness-manifest.md | manifest | YYYY-MM-DD | YYYY-MM-DD |
| scripts/check_harness.py | script | YYYY-MM-DD | YYYY-MM-DD |

## Unmanaged Documentation

| Path | Notes |
|------|-------|
| README.md | Team-maintained project README |

## Staleness Policy

- Default staleness threshold: 30 days
- Architecture docs: refresh when module structure changes
- Manifest: refreshed on every harness-garden audit
- Execution plans: no staleness (they are ephemeral by nature)
```

## Artifact Types

| Type | Description |
|------|-------------|
| `root-agents` | The root AGENTS.md file |
| `module-agents` | A module-level AGENTS.md file |
| `managed-doc` | A documentation file created and maintained by the harness |
| `manifest` | This manifest file itself |
| `script` | A harness script (check_harness.py, etc.) |
| `directory` | A managed directory (exec-plans/active/, etc.) |

## Column Definitions

### Path

Relative path from the repository root. Uses forward slashes (`/`) on all platforms. Directories end with `/` in the Type column but not in the Path column.

### Type

One of the artifact types listed above. Determines how `check_harness.py` validates the entry:

- Files (`root-agents`, `module-agents`, `managed-doc`, `manifest`, `script`): must exist and be non-empty
- Directories (`directory`): must exist as a directory

### Created

ISO date (YYYY-MM-DD) when the artifact was first created by harness-bootstrap. Does not change on subsequent updates.

### Last Verified

ISO date (YYYY-MM-DD) of the most recent verification. Updated by:

- `harness-bootstrap` (on creation)
- `harness-garden` (on audit)
- `check_harness.py` does NOT update this — it only reads

## Unmanaged Documentation

The "Unmanaged Documentation" section records existing team-maintained docs that the harness knows about but does NOT manage. This prevents agents from accidentally overwriting them and documents the relationship between harness and pre-existing docs.

Rules:

- Every existing documentation file discovered during Phase 1.1 of bootstrap should be listed here
- The harness never modifies unmanaged docs
- Unmanaged docs are not checked for staleness
- If an unmanaged doc is later brought under harness management, move it to the "Control Plane Artifacts" table

## Parsing Notes for Scripts

The manifest is designed to be parseable with simple regex. Key patterns:

- Artifact table starts after `## Control Plane Artifacts` and ends at the next `##` heading
- Table rows match: `| <path> | <type> | <date> | <date> |`
- Dates are always YYYY-MM-DD format
- Skip rows containing `---` (separator) or header labels (`Path`, `Type`)

Example regex for extracting artifact rows:

```python
import re
pattern = re.compile(
    r'^\|\s*(.+?)\s*\|\s*([\w-]+)\s*\|\s*(\d{4}-\d{2}-\d{2})\s*\|\s*(\d{4}-\d{2}-\d{2})\s*\|$'
)
```

## Adding New Artifacts

When `harness-feat` or `harness-fix` creates a new managed artifact (e.g., a new execution plan or a new module AGENTS.md), it should add a row to the artifacts table. The entry should use the current date for both Created and Last Verified.

## Removing Artifacts

When an artifact is no longer needed (e.g., a module was deleted), `harness-garden` removes its row from the manifest. Do not leave orphaned entries — they will cause `check_harness.py` failures.
